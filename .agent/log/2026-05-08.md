# 2026-05-08 — Linux migration end-to-end

End-to-end transcript of the Linux migration arc, from plan making
through committed code. Recorded in detail because the bug-finding
process (three review rounds + pristine diff) caught material the
plan didn't anticipate, and a future maintainer redoing this work
would benefit from seeing the iteration.

## Phase 0 — Investigation and v1 plan

User asked for a Linux migration plan, ultrathink-grade. First pass
read every modified-candidate file, the `.vcxproj`, the submodule, and
the configs. Material findings:

- **Codebase is mostly portable already**: no `Windows.h`, no `_MSC_VER`,
  no `__forceinline`/`__declspec`, no Win32 API, no `system()`/`popen`.
  Timing already on `<chrono>`. `Common.h` has `__has_include("cuda_runtime.h")`
  guard. All third-party libs (assimp, eigen, glm, freetype, jsoncpp,
  cli11, imgui, glad, glfw, thrust, OpenMP) are cross-platform.
- **What's actually Windows-shaped**: `.sln`/`.vcxproj` build system;
  ~14 hard-coded `\\` resource path literals; OpenMP flag spelling
  difference; CUDA codegen pinned to `compute_86,sm_86`; vendored Win32
  `libs/`; the `KittenEngine/embree/` directory which isn't actually
  in the `.vcxproj`.
- **Local-box state**: WSL2 Ubuntu 24.04, RTX 5090 (sm_120), nvcc 12.0
  installed (project pins 12.8 — sm_120 needs 12.8+), GCC 13.3,
  `cmake` and `pkg-config` not installed.

v1 plan: 5 phases (env prep, CMakeLists, source patches, build, smoke).

## Phase 1 — First review pass

Sent v1 plan to fresh-Claude (max effort) and Codex (ultrathink) in
parallel. Both converged on the same build-blocking gaps:

- **Missed `fopen_s` × 5 in `Mesh.cpp`** (lines 143, 169, 196, 686, 786).
  MSVC Annex K, won't link on glibc.
- **`CMAKE_CUDA_ARCHITECTURES native` is internally inconsistent** with
  nvcc 12.0 + RTX 5090. `native` would query for sm_120; nvcc 12.0
  doesn't know that arch.
- **OpenMP target-level link missing**: `lbvh.cu` uses
  `#pragma omp parallel for` and `#pragma omp atomic` inside CUDA-compiled
  code. `-Xcompiler=-fopenmp` enables compile but link still needs
  `OpenMP::OpenMP_CXX` propagation.
- **ImGui FetchContent doesn't ship backends**: must explicitly add
  `imgui_impl_glfw.cpp` and `imgui_impl_opengl3.cpp` as TUs and put
  `${imgui_SOURCE_DIR}/backends` on the include path.
- **GLAD ABI**: source uses `<glad/glad.h>` + `gladLoadGLLoader` (glad1).
  Mainline `Dav1dde/glad` defaults to glad2 (`<glad/gl.h>` + `gladLoadGL`).
- **Backslash audit was incomplete**: missed `render.cpp:11–34` (5 sites)
  and runtime concatenations in `Mesh.cpp:284, 306, 311, 328` and
  `KittenAssets.cpp:77`. Pure literal substitution insufficient.
- **Submodule's stripped `Common.h` would silently shadow** if its
  include path got onto `target_include_directories`.

Plus: `frames/` was missing from the resource staging list, jsoncpp
target name varies by distro, no headless library variant exists.

Folded into v2 plan.

## Phase 2 — Auto mode + user additions

User invoked auto mode and gave two specific additions:

1. **Phase 0**: install `git-lfs`, run `git lfs install && git lfs pull`
   for *.bcc files (otherwise GUI launch fails on first scene load with
   a non-obvious error).
2. **Phase 2**: in addition to literal `\\ → /`, change resource-map key
   construction in `KittenAssets.cpp` (`path.string() → path.generic_string()`
   at lines 77, 113-114, 127, etc.) so the rewrite preserves Windows
   compatibility instead of breaking it.

Both folded in. The second was load-bearing — without `generic_string()`,
on Windows the cache stores keys in `\\` form while literal lookups use
`/` form, so every `get<Shader>()` call would silently miss after the
literal rewrite.

## Phase 3 — Implementation

Wrote `CMakeLists.txt` mirroring the `.vcxproj` `<ItemGroup>` exactly
(no globs — would otherwise pull in `embree/MeshCCD.cpp` and the
submodule's standalone `KittenGpuLBVH/main.cpp`). Wrote
`scripts/install-linux-deps.sh` covering apt deps, CUDA 12.8 toolkit
(NVIDIA's wsl-ubuntu repo), git-lfs pull, GLAD generation via Python
`glad` package in a `.venv-glad/` (PEP 668 forces venv on Ubuntu 24.04).

Source patches applied:
- `Mesh.cpp`: 5× `fopen_s → fopen`, runtime `+ "\\"` concat → `(p / x).generic_string()`, map keys → `generic_string()`
- `KittenAssets.cpp`: literal paths + map keys → `generic_string()`
- `Shader.cpp`, `Gizmos.cpp`, `Font.cpp`, `KittenRendering.cpp`, `YarnBall/io/render.cpp`: literal `\\ → /` (17 sites)
- `Algo.h`: 3× `mix(...)` → `glm::mix(...)`
- `StopWatch.cpp`, `Timer.cpp`: `high_resolution_clock::now()` → `steady_clock::now()`

## Phase 4 — First build attempt

`cmake -S . -B build` configured cleanly. `ninja -C build` revealed
issues NOT predicted by the static audit:

### `SymMat.h` anonymous struct in union (compile error)

```
error: member 'glm::vec3 Kitten::hess3::<unnamed union>::<unnamed struct>::diag'
       with constructor not allowed in anonymous aggregate
```

GCC rejects anonymous struct/union members with non-trivial constructors.
GLM's `vec3` has user-defined ctors. `-fms-extensions` does NOT relax
this (it allows anon structs but explicitly disallows ctors in them).
Refactor: name the inner struct (`struct s_t { vec3 diag; vec3 upper; } s;`),
update access from `H.diag` to `H.s.diag` at 4 sites in `cosserat.cu`.
Memory layout preserved.

### `StopWatch.cpp` / `Timer.cpp` chrono mismatch

```
no match for 'operator-' (operand types are
  'std::chrono::time_point<system_clock,...>' and
  'std::chrono::time_point<steady_clock,...>')
```

`high_resolution_clock` is an alias for `steady_clock` on MSVC, but for
`system_clock` on libstdc++. Header field is `steady_clock::time_point`.
Fix: use `steady_clock::now()` consistently in the .cpp files.

### `Algo.h::mix(double, double, double)` lookup failure

```
no matching function for call to 'mix(const double&, const double&, const double&)'
candidate: 'template<int s, int c, class T> ... Kitten::mix(mat<...>, mat<...>, T)'
```

`Common.h` declares `Kitten::mix(mat, mat, T)`. Inside template
functions in `namespace Kitten`, the unqualified `mix(double, double, double)`
resolves only to `Kitten::mix` (directly-declared overload). MSVC's
lookup happens to find `glm::mix` via the using-directive too; GCC's
stricter two-phase lookup doesn't. Fix: qualify as `glm::mix(...)` at
3 sites.

### CUDA arch detection

`if(NOT DEFINED CMAKE_CUDA_ARCHITECTURES)` was always-false. CMake
policy CMP0104 auto-initializes to `52` during `project(... LANGUAGES CUDA)`.
First sentinel attempt: `STREQUAL "52"` (later refined). Auto-detect via
`${CUDAToolkit_BIN_DIR}/__nvcc_device_query` returns `120` on this 5090.

### Build succeeded; headless worked

Headless letterS: `Total verts: 32931`, sim/real 2.5×, exit 0, 4 OBJs.
Headless cable_work_pattern --twist: 31 frames, sim/real 1.7×, exit 0.

### GUI: mostly worked but shader includes failed

Window opened but every shader compile failed:

```
err: shader compilation failed.
0:3(1): preprocessor error: include "kittenCommonVert.glsl" not found
```

Root cause: `KittenPreprocessor.cpp:65` regex `^ *#include +...` lacks
the `multiline` flag. With C++17 default ECMAScript flavor, `^` only
matches offset 0. Shaders start with `#version`, so the regex never
matched any line and the engine's `#include` resolution was a silent
no-op. **This was a latent bug on Windows too** — Windows GL drivers
must have been tolerating unresolved `#include` directives somehow,
or always failing silently.

Fix: `regex pattern("^ ...", regex::ECMAScript | regex::multiline);`.

GUI mode also needed GL 4.6 → 4.5 fallback for WSLg's Mesa-on-D3D12
(caps at 4.5) and a 16xMSAA → 0xMSAA retry path.

After all fixes: GUI opens, all shaders compile, no errors. Window
stable for >12s.

## Phase 5 — Second review pass

Round-2 Claude + Codex reviews. New findings:

- **`Font.cpp:188` map key still uses `path.string()`** — Windows
  cache writes `\\` form, lookup queries `/` form, silent miss.
  Fix: `generic_string()`.
- **`Mesh.cpp` Assimp texture paths**: a `.obj` with
  `map_Kd textures\foo.png` returns `aiString` with `\` even on Linux.
  `(parent / "textures\foo.png").generic_string()` treats the RHS as
  one filename. Fix: `std::replace(s.begin(), s.end(), '\\', '/')`
  before joining.
- **`third_party/glad/` ignored but CMake required it** — Codex
  flagged: a fresh clone breaks. Replaced `FATAL_ERROR` with CMake
  auto-bootstrap (creates venv, runs `glad`, atomic rename into place).
- **`scripts/install-linux-deps.sh` missing `wget`** in step 1 (used
  by step 3); CUDA URL hardcoded to `wsl-ubuntu` even on native Linux
  → switch to `grep -qiE "(microsoft|wsl)" /proc/version`.
- **GLAD generation not atomic** → `mktemp -d` + `mv`, with the temp
  dir on the same filesystem as the destination.
- **`Sim::advance` UB** — declared `float`, no return statement.
  Briefly changed to `void` (later reverted, see Phase 6).
- **CMakeLists `STREQUAL "52"` overrides legitimate `-D=52`** —
  Codex caught the edge case. Refined to cache TYPE probe.
- **`file(RENAME)` cross-filesystem** — Codex caught: `_glad_tmp`
  under `${CMAKE_BINARY_DIR}` may be on tmpfs while
  `${CMAKE_SOURCE_DIR}/third_party` is on disk. Same-filesystem
  staging fixes it.
- **Many `cudaMalloc`/`cudaMemcpy*`/`cudaStream*` returns unchecked**
  — pre-existing, deferred.
- **`Sim::step(float h)` ignores `h`** (calls `advance(maxH)`) — pre-existing,
  deferred.

Each acted on. Re-tested all green.

## Phase 6 — Audit against pristine `../YarnBall_2`

User asked to check for over-engineering and verify behavior preservation
against the pristine repo. `diff -u` of every modified file against
`/home/daizi/Research/YarnBall_2/`. Found:

1. **`Sim::advance` signature** had been changed to `void`. Reverted to
   `float advance(float)` returning 0 — public-API change is beyond
   migration scope.
2. **`Mesh.cpp:336` printf** had been swept by the literal rewrite:
   `printf("...sub-mesh %s (\\%s)\n", path.string()...)` →
   `printf("...sub-mesh %s (/%s)\n", path.generic_string()...)`. Plan
   had explicitly noted this was a log message, not a path. Reverted.
3. **`KittenInit.cpp` GL version**: had been lowered to 4.5 default with
   MSAA fallback. Reverted to original 4.6 + 16xMSAA primary, falling
   back to 4.5 + 0xMSAA only if primary fails. Native Windows now gets
   the original context exactly.
4. **`CMakeLists.txt`**: had `CUDA_SEPARABLE_COMPILATION ON` +
   `CUDA_RESOLVE_DEVICE_SYMBOLS ON` adding `-rdc=true` (not in original
   .vcxproj, no cross-TU `__device__` calls exist), and
   `CMAKE_POSITION_INDEPENDENT_CODE ON` (irrelevant for static lib).
   Removed all three.
5. **`scripts/install-linux-deps.sh` `--no-cuda` flag**: redundant with
   the auto-skip already there. Removed.

After reverts, the diff per file is strictly minimal — every line
change traces to a specific GCC/Linux/WSLg requirement. Nothing
speculative.

## Phase 9 — GUI crash on WSLg → CUDA-GL interop bypass

User then ran:

```
./Gui configs/cable_work_pattern.json --twist -s -e -n 30 --exit -o /tmp/twist_
```

(Without `--headless`.) Crash:

```
Total verts: 65065
GPUassert: context is destroyed lbvh.cu:337
```

This was the §7.1 risk we'd documented but not solved: WSLg's GL stack
(Mesa software / D3D12) doesn't expose NVIDIA's CUDA-GL interop bridge.
`cudaGraphicsGLRegisterBuffer` destroys the CUDA context, surfacing
later as `"context is destroyed"` errors in unrelated CUDA calls.

Probed the renderer string: `Mesa | llvmpipe (LLVM 20.1.2, 256 bits) | 4.5 (Core Profile)`
— pure software rasterizer, definitely no NV interop.

User asked for a way to make GUI work on this setup. Discussed three
options: native Windows (no code change), headless workflow (works
already), or a runtime CUDA-GL bypass (~30 line patch).

Implementation:
- **`ComputeBuffer.h`**: added `cudaGLInteropBroken()` — probes
  `glGetString(GL_RENDERER)` once, caches result. Matches `"D3D12"`
  or `"llvmpipe"` substrings → set bypass flag.
- **`cudaWriteGL` / `cudaReadGL`**: branch on the flag. Bypass path:
  `cudaMemcpy DtoH` to a `thread_local std::vector<uint8_t>` then
  `glBufferSubData`, and the symmetric direction for read.
- **`KittenInit.cpp`**: `printf("GL: %s | %s | %s\n", VENDOR, RENDERER, VERSION)`
  at startup so the active stack is visible.

**Zero behavior change on native NVIDIA**: the renderer string doesn't
match either substring, the bypass flag stays false, the existing fast
interop path runs unchanged. Cost on the bypass path: ~1–2 ms/frame at
65k verts (cudaMemcpy DtoH + glBufferSubData), negligible at 30 FPS.
Visualization is still software-rasterizer-bound on llvmpipe (~10–15 FPS)
but the simulator stays GPU-fast.

Verified:
- GUI on WSLg llvmpipe with `--twist -s`: 12+ seconds stable, 0 CUDA
  errors, 0 shader errors.
- Headless still works at 1.7×.

Second commit: `4c92ced` "Bypass CUDA-GL interop on WSLg + add Linux
run guide".

## Phase 10 — cwd-relative path UX trip

User then tried running from the repo root (not `build/`):

```

filesystem error: recursive directory iterator cannot open directory: KittenEngine/shaders
```

Engine asset paths are cwd-relative. Original Windows behavior: Visual
Studio's debugger silently sets CWD to `$(OutDir)`. Linux: user must
`cd build` themselves. Considered adding `chdir(canonical("/proc/self/exe").parent_path())`
at startup but rejected — would change resolution of user-supplied
relative `--output` paths, which is a behavior change. Accepting the
papercut.

## Open threads (pre-existing, deferred)

Each was flagged during review but explicitly out of scope for the
migration commit. Touching any of them is its own task with its own
commit:

- `Sim::step(float h)` ignores `h` (calls `advance(maxH)`); should be
  `advance(h)`.
- Many CUDA call returns unchecked (`cudaMalloc`, `cudaMemcpy*`,
  `cudaStream*`, `cudaGraph*` in `step.cpp` and `YarnBall.cu`). Code
  relies on a single `cudaGetLastError()` at the end.
- `KittenPreprocessor.cpp:65` regex matches `#include` inside block
  comments — would substitute commented-out includes. Latent on
  Windows too.
- `jsonBuilder.cpp::buildFromJSON` resolves `curveFile` paths against
  process CWD instead of the config file's directory.
- `configs/relax0.json` and `relax1.json` reference `.bcc` files that
  don't exist in the repo.
- No headless library variant — `YarnBall.h` transitively pulls
  GLAD/GLFW/ImGui via `KittenEngine.h`.
- `Mesh.cpp::loadMeshFrom` material loop: `(Texture*)resources[p]`
  after a failed `loadTexture(p)` inserts a permanent `nullptr` cache
  entry, masking subsequent successful loads of the same key.
- `main.cpp:216` `if (ImGui::TreeNode("Export"))` block has no
  matching `ImGui::TreePop()` — pre-existing ImGui spam in GUI logs.
- Engine asset paths are cwd-relative; `cd build/` is required.

## Lessons for next migration

- **Three review rounds found real bugs each round**, including in
  the previous round's fixes. Iterating fresh-Claude + Codex with
  delta-only briefs is high yield.
- **Pristine-vs-current diff is mandatory** at the end. Reviewers
  validate the changes you have, not the changes you should have.
  An audit against the original surfaces over-engineering they
  miss.
- **Build doesn't catch design**: GCC strictness (anon-aggregate,
  two-phase lookup), latent regex bugs, runtime CUDA-GL interop
  failures only show up at compile / first run.
- **Auto-detect at runtime over compile-time flags** for environment
  differences (CUDA arch, GL stack capabilities). Less surface area
  for users to misconfigure, less code to maintain across platforms.
